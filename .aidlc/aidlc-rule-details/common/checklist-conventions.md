# Checklist Conventions

**Purpose**: Define the universal format for the checklist artifact that **every AI-DLC stage** produces. Checklists are the primary mechanism that prevents items from being missed.

---

## Why Checklists

A 40-person company iterating fast cannot rely on the AI's memory or the pod's vigilance alone. A written checklist:
- Makes "what we agreed" inspectable in code review
- Makes "what was skipped" explicit and auditable
- Prevents silent omissions across stack handoffs (FE / BE / Mobile)

Every stage MUST produce a checklist artifact. No exceptions.

---

## Checklist File Naming

Each stage produces a `*-checklist.md` file in its stage output directory. Examples:

| Stage | Checklist file |
|-------|----------------|
| Business Requirements | `aidlc-docs/inception/business-requirements/business-requirements-checklist.md` |
| Design Intake | `aidlc-docs/inception/design/design-intake-checklist.md` |
| Reverse Engineering | `aidlc-docs/inception/reverse-engineering/reverse-engineering-checklist.md` |
| Requirements Analysis | `aidlc-docs/inception/requirements/requirements-checklist.md` |
| Workflow Planning | `aidlc-docs/inception/plans/workflow-planning-checklist.md` |
| Functional Design (per unit) | `aidlc-docs/construction/{unit}/functional-design/functional-design-checklist.md` |
| Code Generation (per unit) | `aidlc-docs/construction/{unit}/code/code-generation-checklist.md` |
| Grill-Me #1 (per unit) | `aidlc-docs/construction/{unit}/grill-me-1/{unit}-grill-me-1-checklist.md` |
| Manual QA (per unit) | `aidlc-docs/construction/{unit}/manual-qa/{unit}-manual-qa-checklist.md` |
| Grill-Me #2 (per unit) | `aidlc-docs/construction/{unit}/grill-me-2/{unit}-grill-me-2-checklist.md` |
| Build & Test | `aidlc-docs/construction/build-and-test/build-and-test-checklist.md` |
| Production Readiness | `aidlc-docs/operations/production-readiness/production-readiness-checklist.md` |

(All other stages follow the same `<stage-name>-checklist.md` pattern in their stage directory.)

---

## Checklist Item States

Each item is one of three states:

| Marker | Meaning |
|--------|---------|
| `- [ ]` | Pending — not yet addressed |
| `- [x]` | Done — addressed and confirmed |
| `- [~] N/A: <reason>` | Not applicable — reason mandatory |

A stage CANNOT complete while any item is `- [ ]`. The AI must convert every item to `- [x]` or `- [~]` before advancing.

---

## Checklist File Template

```markdown
# <Stage Name> Checklist — <Project / Unit Name>

**Tier**: <greenfield | feature | bugfix>
**Generated by**: AI-DLC
**Generated at**: <ISO timestamp>
**Last updated**: <ISO timestamp>

## How to use this checklist

- Each item is `- [ ]`, `- [x]`, or `- [~] N/A: <reason>`
- The AI proposes the initial list; the pod may add/remove items before signing
- A stage cannot pass its gate while any `- [ ]` items remain
- N/A reasons are reviewed at sign-off — bad reasons block sign-off

## Items

### Section 1 — <e.g., Problem definition>

- [ ] Item 1 — <description>
- [ ] Item 2 — <description>

### Section 2 — <e.g., Scope>

- [ ] Item 3 — <description>
- [ ] Item 4 — <description>

<...more sections...>

## Pod Annotations

<optional notes from Tech Lead or Dev — e.g., "Item 7 N/A because we run on-prem only">

## Modification Log

| Timestamp (ISO) | Editor | Change |
|-----------------|--------|--------|
| <ts> | <name> | Marked Item 3 [x] |
| <ts> | <name> | Added Item 9 |
| <ts> | <name> | Marked Item 5 [~] N/A: no payments in scope |
```

## Item Quality Rules

A good checklist item:
1. **Is binary** — can be answered yes / no / N/A. No vague items like "consider performance"
2. **Names the artifact** — e.g., "API contract documented in `requirements.md` § 3.2", not "API documented somewhere"
3. **Names the owner role** — e.g., "(Tech Lead) Confirm cloud target is team-approved"
4. **Is independent** — items that cascade should be split: "Database chosen" + "Database connection string in env example"

The AI generates an initial list and the pod may edit before signing. Pod additions are appended; pod removals require an N/A entry with reason rather than deletion.

---

## N/A Reason Quality

Acceptable N/A reasons:
- "Project has no payments — PCI compliance not in scope"
- "Mobile-only project — no FE web bundle to optimize"
- "Bugfix Tier — full traceability matrix not required"

Unacceptable N/A reasons (the AI MUST reject these and re-prompt):
- "N/A"
- "Not needed"
- "Skip"
- "<empty>"

If the pod insists on a vague N/A reason, the AI logs the dispute in `audit.md` and refuses to advance the gate until the reason is specific.

---

## Tiered Checklist Sizes (BR stage only)

The BR stage uses three different baseline checklist templates by Tier — see `inception/business-requirements.md`. Other stages do not vary checklist size by Tier; they vary depth instead via `depth-levels.md`.

---

## Cross-Stage Linking

When an item references another stage's artifact, the link MUST be a relative path:

```markdown
- [x] User personas captured — see `../user-stories/personas.md` § "Persona 2"
```

Broken cross-stage links are a `content-validation.md` violation.

---

## Anti-patterns

- ❌ Stage advancing while items are `- [ ]`
- ❌ Empty / vague N/A reasons
- ❌ Items that overlap or restate each other
- ❌ Items that ask the user to "consider X" without a yes/no outcome
- ❌ Deleting items instead of marking them N/A (kills the audit trail)
