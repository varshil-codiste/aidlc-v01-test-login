# Approval Gates

**Purpose**: Codify the FIVE written sign-off gates that block workflow progression in AI-DLC. Gates are mechanical: a file with the required signatures must exist before the AI advances.

---

## Gate Inventory

| # | Gate | Trigger | Signoff file path | Signers |
|---|------|---------|-------------------|---------|
| 1 | **Business Requirements** | After BR checklist passes | `aidlc-docs/inception/business-requirements/business-requirements-signoff.md` | Tech Lead + Dev |
| 2 | **Workflow Plan** | After Workflow Planning produces `execution-plan.md` | `aidlc-docs/inception/plans/execution-plan-signoff.md` | Tech Lead + Dev |
| 3 | **Code Generation** (per UoW) | After Code Generation Part 1 (Planning) produces the per-unit codegen plan | `aidlc-docs/construction/{unit-name}/code/{unit-name}-codegen-signoff.md` | Tech Lead + Dev |
| 4 | **Code Review** (per UoW) | After Code Review runs lint + security + tests + AI review on generated code | `aidlc-docs/construction/{unit-name}/code-review/{unit-name}-code-review-signoff.md` | AI-DLC verdict + Tech Lead + Dev |
| 5 | **Production Readiness** | After Production Readiness checklist passes | `aidlc-docs/operations/production-readiness/production-readiness-signoff.md` | Tech Lead + Dev |

## Universal Signoff File Template

Every signoff file uses this exact structure. The AI generates the file pre-filled with everything except the two signature lines.

```markdown
# Signoff — <Gate Name> — <Project / Unit Name>

**Tier**: <greenfield | feature | bugfix>
**Generated by**: AI-DLC
**Generated at**: <ISO timestamp>

## What Is Being Approved

<one-paragraph summary written by the AI, listing the artifact files this signoff covers>

## Artifacts Referenced

- <relative path 1>
- <relative path 2>
- ...

## Compliance Summary (if any extensions enabled)

| Extension | Rule | Status | Notes |
|-----------|------|--------|-------|
| Security  | SECURITY-01 | Compliant / Non-compliant / N/A | <rationale> |
| ...       | ...         | ...                              | ... |

## Open Risks / Caveats

<list any risks the AI flagged that the pod must accept by signing, or write "None">

## AI-DLC Code Review Verdict (Gate #4 only — omit for other gates)

<see Gate #4 section below for the full verdict-block template>

## Signatures

- [ ] Tech Lead: ____________________  Date: __________  (ISO 8601)
- [ ] Dev:       ____________________  Date: __________  (ISO 8601)

## Substitution (only if applicable)

- [ ] Substitute for: <Tech Lead | Dev> — Substitute name: ____________________
      Reason primary unavailable: ____________________
      Authorized by pod.md substitute roster: <yes | no>

## Objection (only if applicable)

<one of the two signers writes their objection here instead of signing>

## Resolution (only if Objection was used)

<resolution text + both signers re-affirm by ticking their boxes above>
```

---

## Validation Rules

Before the AI may advance past a gate, it MUST verify the signoff file:

1. **File exists** at the canonical path for the gate
2. **Both signature checkboxes are `[x]`** (not `[ ]`)
3. **Both name fields are non-empty** and resolve to entries in `pod.md` (or to a registered substitute)
4. **Both date fields are valid ISO 8601** dates within the last 30 days (older signatures are stale)
5. **No `## Objection` section is present and unresolved**
6. **Compliance Summary table has no `Non-compliant` rows** for opted-in extensions (Compliant or N/A only)

If any check fails, the AI:
- Logs the failure in `audit.md`
- Posts a clear blocking message to the user identifying which check failed
- Refuses to advance

## Per-Gate Specifics

### Gate #1 — Business Requirements

Triggered after `inception/business-requirements.md` finishes the checklist.

**Artifacts referenced**:
- `business-requirements.md`
- `tier.md`
- `business-requirements-checklist.md`
- All files under `business-requirements/sources/`

**Compliance summary**: Empty for Gate #1 (extensions are not yet opted-in).

### Gate #2 — Workflow Plan

Triggered after `inception/workflow-planning.md` produces `execution-plan.md`.

**Artifacts referenced**:
- `execution-plan.md`
- `execution-plan-mermaid.md` (if separate)

**Compliance summary**: Lists each opted-in extension and confirms its rules are scheduled to run at the appropriate stages.

### Gate #3 — Code Generation (per UoW)

Triggered after `construction/code-generation.md` Part 1 produces `{unit-name}-code-generation-plan.md`.

**Artifacts referenced**:
- `{unit-name}-code-generation-plan.md`
- All design artifacts for this unit

**Compliance summary**: Mandatory — lists every applicable extension rule and its compliance status for the planned code.

**Critical**: This gate is **per-unit**. A multi-unit project has one Gate #3 signoff file per unit.

### Gate #4 — Code Review (per UoW)

Triggered after `construction/code-review.md` finishes the four mandatory checks on generated code.

**Artifacts referenced**:
- `{unit-name}-code-review-report.md` — full review output
- `{unit-name}-lint-report.md` — linter output
- `{unit-name}-security-report.md` — SAST + dependency-scan output
- `{unit-name}-test-report.md` — unit test results + coverage
- `{unit-name}-ai-review.md` — AI's structured review
- `{unit-name}-manual-qa-checklist.md` — scenarios + bugs (Stage 14)
- `{unit-name}-manual-qa-results.md` — Manual QA summary with verdict
- `{unit-name}-grill-me-2-results.md` — `/grill-me-2` quiz score + per-question verdicts

**AI-DLC Verdict** (mandatory in this signoff file):

```markdown
## AI-DLC Code Review Verdict

**Verdict**: <PROCEED | BLOCK>
**Generated at**: <ISO timestamp>
**Reviewing model**: <model name and version>

| Check | Result | Notes |
|-------|--------|-------|
| Linting | ✅ Pass / ❌ Fail | <error count, warning count> |
| Code Security (SAST) | ✅ Pass / ❌ Fail | <vulnerability count by severity> |
| Test Results | ✅ Pass / ❌ Fail | <pass / fail / coverage %> |
| AI Review | ✅ Approve / ⚠️ Concerns / ❌ Reject | <summary> |
| **Manual QA** (per UoW, Stage 14) | ✅ All-PASS / ❌ Fail (open bugs) | <PASS count / FAIL count / open-bug count / iteration #> |
| **Grill-Me #2** (per UoW) | ✅ PASS (≥0.85) / ❌ FAIL (<0.85) | <score = n/total; loopback option if FAIL> |

**Verdict rationale**: <one paragraph explaining the verdict>

**If BLOCK** — required actions before re-review:
1. <action 1>
2. <action 2>
...
```

**Compliance summary**: Mandatory — every opted-in extension rule must be Compliant or N/A. A single Non-compliant row forces the verdict to BLOCK.

**Verdict semantics**:

| 4 auto checks | Manual QA (Stage 14) | Grill-Me #2 | Verdict |
|---|---|---|---|
| all ✅ | ✅ All-PASS | ✅ PASS | **PROCEED** (pod may countersign → advance to Build & Test, Stage 15) |
| all ✅ | ✅ All-PASS | ❌ FAIL | **BLOCK** (run `/grill-me-2` loopback path) |
| all ✅ | ❌ Fail (open bugs) | * | **BLOCK** (re-enter Stage 12b for bug fixes, then Stage 13, then Stage 14 again) |
| any ❌ | * | * | **BLOCK** (re-enter Stage 12b for code findings) |

The four auto-check rows are emitted at the end of Stage 13 (Code Review). The Manual QA row is filled by Stage 14 (`construction/manual-qa.md`). The Grill-Me #2 row is filled by the `/grill-me-2` sub-ritual (`construction/grill-me-2.md`). Pod countersignature is collected only after all six rows are green.

**Pod countersignature** is required even when the four auto-checks are PROCEED + Manual QA all-PASS + Grill-Me #2 PASS — the pod's signature represents the final human accountability. If the pod believes the code is wrong despite all six rows green, they refuse to sign and write the issue under `## Pod Override`.

**Critical**: This gate is **per-unit**. A multi-unit project has one Gate #4 signoff file per unit.

### Gate #5 — Production Readiness

Triggered after `operations/production-readiness.md` produces the readiness checklist + runbook.

**Artifacts referenced**:
- `production-readiness-checklist.md`
- `runbook.md`
- `rollback-plan.md`
- `monitoring-dashboard-links.md` (if Observability extension ran)

**Compliance summary**: Final cross-check of all opted-in extensions.

## Bugfix Tier — Light-form Combined Sign-off

For bugfix Tier with single-file scope, Gates #2 and #5 may be combined into the BR signoff. See `tiered-mode.md` for the exact light-form template. Gates #1, #3, and #4 are NEVER skippable.

---

## Severity-1 Hotfix Exception

For bugfix Tier marked severity-1:

- Gate #1 and Gate #3 may be signed by Tech Lead alone
- Gate #4 still requires the full AI-DLC verdict block (all six rows); Tech Lead may countersign alone for the pod-signature lines (Dev countersignature retroactive within 24 hours)
- Dev signature must be added retroactively within 24 hours for all gates
- The signoff file MUST have a `## Severity-1 Exception` section explaining the hotfix scope
- The retroactive Dev signature is logged in `audit.md` with timestamp

This exception is the only situation where a single human signature unblocks a gate. The Code Review **AI verdict** is NEVER skippable — even sev-1 must run lint/security/tests. **Manual QA (Validation Rule 7) and Grill-Me #2 (Validation Rule 8) are ALSO NEVER skippable under sev-1** — both Tech Lead and Dev must execute Manual QA scenarios and pass Grill-Me #2 before any countersign. Tech-Lead-solo Grill-Me #2 is forbidden at Gate #4 because the quiz tests collective understanding, not just one signer's.

---

## AI-DLC Verdict Validation (Gate #4 only)

For Gate #4, before the pod may countersign, the AI-DLC verdict block MUST validate:

1. The verdict line says exactly `PROCEED` or `BLOCK` (no other strings)
2. All six check rows have ✅ or ❌ explicitly (no blanks) — four auto-checks + Manual QA + Grill-Me #2
3. If verdict is `PROCEED`, no row is ❌
4. If verdict is `BLOCK`, the "required actions before re-review" list is non-empty
5. The reviewing model name and ISO timestamp are present
6. The verdict was generated within the last 7 days (older verdicts must be regenerated since the code may have changed)
7. **Manual QA results file** (`{unit-name}-manual-qa-results.md`) exists, has zero `❌ FAIL` scenarios, and zero bugs with status `OPEN` or `IN-PROGRESS`. Iteration count ≤ 3.
8. **Grill-Me #2 results file** (`{unit-name}-grill-me-2-results.md`) exists, has Verdict = `PASS` (aggregate ≥ 0.85), and Generated-at is within the last 7 days.

Rules 7 and 8 are NEVER skippable, including under the Severity-1 Hotfix Exception. The human attestation (Manual QA) and the read-back quiz (Grill-Me #2) are the team-specific safety net that compensates for what the AI checks cannot catch — relaxing them under any exception defeats their purpose.

If validation fails, the AI regenerates the verdict block (or directs the pod to re-run Stage 14 / `/grill-me-2`). If pod countersignatures already exist on a stale verdict, they are invalidated and must be re-collected after regeneration.

---

## Anti-patterns

- ❌ Editing a signoff file to flip `[ ]` to `[x]` without a name and date
- ❌ Re-using last week's signoff for a new artifact set (date check fails)
- ❌ Signing a gate while a `## Objection` is unresolved
- ❌ Combining Gates #1 + #3 — never permitted (#1 and #3 cover different artifacts)
- ❌ Pod countersigning a `BLOCK` verdict to force progression (must address findings instead)
- ❌ AI editing its own verdict from `BLOCK` to `PROCEED` without re-running the checks
- ❌ AI advancing because "the signoff was discussed in chat" — only file presence counts
