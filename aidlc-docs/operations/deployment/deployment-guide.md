## What this app is

A small login + account-setup feature with merchant/seller role capture. One backend (NestJS) talks to one Postgres. One frontend (Next.js) renders the public routes.

## Topology

```
[browser]                                          
   │  https://app.local                            
   ▼                                               
[frontend container]  next start -p 3000          
   │  /api/* rewrite → http://backend:4000        
   ▼                                               
[backend container]  node dist/.../main.js  :4000  
   │                                               
   ▼                                               
[postgres 16]  db:5432 internal / 5433 host        
```

## Required environment variables (all containers)

| Var | Container | Notes |
|-----|-----------|-------|
| `DATABASE_URL` | backend | `postgresql://USER:PASS@HOST:5432/DB?schema=public` |
| `JWT_PRIVATE_KEY` | backend | PEM, RS256-2048; generated by `bash scripts/gen-keys.sh` |
| `JWT_PUBLIC_KEY` | backend | PEM, RS256-2048 |
| `FE_ORIGIN` | backend | exact origin for CORS — `https://app.local` in prod, `http://localhost:3000` in dev |
| `APP_ENV` | backend | `prod` / `ci` / `dev` — toggles `Secure` cookie flag |
| `LOG_LEVEL` | backend | `info` (default) |
| `PORT` | backend | `4000` |
| `NEXT_PUBLIC_API_BASE_URL` | frontend | `https://app.local` (FE proxies `/api/*` to it) |

Missing required vars → `assertEnv()` exits the BE with code 1 BEFORE the HTTP server starts (NFR-REL-003).

## Initial deploy (greenfield)

```bash
# 1. Provision Postgres 16
#    Create database `auth` with role `app` (or your own — set DATABASE_URL accordingly).

# 2. Generate JWT keys ONCE per environment and store them in your secret manager
bash scripts/gen-keys.sh   # writes JWT_PRIVATE_KEY + JWT_PUBLIC_KEY into .env.local

# 3. Apply Prisma migrations against the empty DB
DATABASE_URL=<prod-url> npx prisma migrate deploy
#  → applies 0001_init AND 0002_add_role

# 4. Build container images
docker build -t auth-backend  apps/backend/
docker build -t auth-frontend apps/frontend/

# 5. Run, passing the env vars into the containers
docker run -e DATABASE_URL=... -e JWT_PRIVATE_KEY=... -e JWT_PUBLIC_KEY=... \
           -e FE_ORIGIN=https://app.local -e APP_ENV=prod -e PORT=4000 -p 4000:4000 auth-backend
docker run -e NEXT_PUBLIC_API_BASE_URL=https://app.local -p 3000:3000 auth-frontend
```

## Incremental deploy — `roles-profile` UoW additions

See also `roles-profile-deployment-amendment.md`. The relevant pieces in-line:

- **Migration `0002_add_role`** is **backwards-compatible**: adds Postgres `Role` enum + `users.role NOT NULL DEFAULT 'SELLER'`. All pre-existing rows backfill to `SELLER` in a single `ALTER TABLE`. No downtime, no separate backfill job.
- **Rollout order (recommended FE-first)**:
  1. `npx prisma migrate deploy` (applies `0002_add_role`).
  2. Deploy new FE image (sends `role` on signup; old BE silently ignores it).
  3. Deploy new BE image (enforces `role` ∈ `{MERCHANT, SELLER}` on signup; serves `role` on every response).
- **Rollback**: keep the migration applied; roll BE + FE images back. Dropping the column is destructive for post-migration signups and is **NOT** part of the rollback plan. Manual destructive rollback SQL exists at `apps/backend/prisma/migrations/0002_add_role/migration-rollback.sql` for cold environments only.

## Post-deploy verification (per release)

```bash
# Health probe
curl https://app.local/api/health                # expect 200 {status:"ok"}
curl https://app.local/api/.well-known/jwks.json # expect 200 with `keys`

# Smoke signup (Merchant)
curl -s -X POST https://app.local/api/auth/signup \
  -H 'content-type: application/json' \
  -d '{"email":"smoke+m+'$(date +%s)'@example.com","displayName":"Smoke M","password":"CorrectHorseBattery42!","role":"MERCHANT"}' \
  | jq

# Smoke negative — invalid role
curl -s -o /dev/null -w "%{http_code}\n" -X POST https://app.local/api/auth/signup \
  -H 'content-type: application/json' \
  -d '{"email":"a@b.co","displayName":"X","password":"CorrectHorseBattery42!","role":"ADMIN"}'
# expect 400
```

## Cookies — environment-aware

- `APP_ENV=prod` / `ci` → cookies marked `Secure` (HTTPS only).
- `APP_ENV=dev` → `Secure` off so localhost HTTP works.
- Both modes keep `HttpOnly` + `SameSite=Lax` + `Path=/`.

## Logs

Backend writes structured JSON to stdout (pino). Sensitive fields (`password`, `*token*`, `cookie`, `authorization`, `$argon2id$`, JWT prefix `eyJ…`) are redacted automatically (NFR-S07, BR-A12). No new fields introduced by `roles-profile`.

## Secrets

- `JWT_PRIVATE_KEY` + `JWT_PUBLIC_KEY` — RSA-2048 PEM. Rotate annually OR if compromise suspected. Procedure: generate new keys, deploy with BOTH old and new public keys advertised at JWKS (so tokens signed with the old key still verify), wait out the refresh-token TTL (7 days), retire old keys. (v1 has a single-key flow; multi-key JWKS is a Stage-18 follow-up.)
- DB password — stored in your secret manager; never committed.

## Backup + restore (Postgres)

```bash
# Backup
pg_dump -h $PGHOST -U app -d auth -F c -f auth-$(date +%Y%m%d).dump

# Restore (DB must be empty)
pg_restore -h $PGHOST -U app -d auth auth-YYYYMMDD.dump
DATABASE_URL=... npx prisma migrate deploy   # bring schema to latest
```

## Known caveats (carried from cycle 1)

1. **High-severity transitive prod-dep vulns** in multer / next / lodash / js-yaml / @nestjs/core. Auth UoW has zero `multer` runtime exposure (no `@UploadedFile()` anywhere). Fix path: NestJS 10 → 11 major bump — scheduled for **before any production deploy** (Stage-18 / Gate-#5 precondition).
2. **Single-process rate-limit map** — fine at ≤ 50 users. To scale to multiple BE replicas, swap `Map<email, timestamp[]>` for Redis sorted-set with same key/value semantics; no API-surface change.
3. **No password-reset / no MFA / no OAuth** — intentionally out of scope per BR § 1.4. Add these via follow-up UoWs.
